## What Changed in v4.3.5

### 1. Tightened CI Proximity Search
- **Window reduced**: 500 chars -> 150 chars
- **CI label gate**: Unlabeled patterns (parentheses, brackets) only fire if "CI" or "95% CI" appears in context
- **Negative validation**: Rejects CI if a different effect value is closer to it
- **Removed hyphen_range pattern**: Too many false positives (matched mean+SD as CI)
- **Result**: 52 CIs recovered (vs 43 with old settings on curated corpus)
  - 9 false positives eliminated

### 2. Corpus Curation
- **15 non-RCT papers excluded**: Meta-analyses, post-hoc analyses, protocols, reviews
- **43 confirmed RCT papers retained**: 100% extraction rate
- **10 papers designated as held-out test set** (deterministic seed=42)

### 3. Text Preprocessor Integrated
- Unicode normalization (ligatures, dashes, quotes, spaces)
- Dehyphenation (line-break hyphen removal)
- Column reordering (two-column layout detection)
- Header/footer removal
- Section detection (abstract, methods, results, discussion)
- **Performance fix**: O(n^2) bug in _build_blocks fixed to O(n)

### 4. Validated RCT PDFs Downloaded
- **15 validated PDFs** from external validation dataset (8 cached + 7 FTP downloads)
- Trials: DAPA-HF, EMPEROR-Reduced, PARADIGM-HF, EMPA-REG, CANVAS, LEADER, FOURIER, SELECT, etc.
- Ground truth available for 13 of these trials

## Next Steps

### P0: Fix Validated PDF Extraction
The 11 validated PDFs with 0 extractions are the biggest gap. These are prestigious
journal papers (NEJM, Lancet) with layouts the extractor can't handle:
1. Debug what text pdfplumber extracts from these PDFs
2. Identify if it's a parsing issue or pattern issue
3. Consider journal-specific layout handling

### P1: Improve Ground Truth Matching
Current matching is too strict and doesn't account for:
- Papers with many effects (need primary outcome identification)
- Effect type variations (HR vs hazard ratio)
- Papers reporting secondary endpoints that differ from GT primary

### P2: Address Remaining 17 Missing CIs
These are genuinely missing from text (not false negatives from search):
- 36 were IN_TABLE format (before curation)
- Consider table extraction as separate module

### P3: Regression Tests
- Create pytest suite that validates baseline metrics
- Alert if CI completion drops below 85%
- Track per-PDF extraction stability

## Lessons Learned

1. **Verification before celebration**: v4.3.4's 92.3% was inflated by ~20% false positives
2. **Window size matters**: 500 chars catches wrong CIs; 150 chars is the sweet spot
3. **CI label is critical**: Without "95% CI" or "CI" nearby, parenthesized ranges are ambiguous
4. **Negative validation works**: Checking if CI is closer to a different value catches cross-matching
5. **Performance matters**: O(n^2) in preprocessor caused infinite-seeming hangs on large PDFs
6. **OA corpus != major journals**: Our 43 curated OA papers extract well but NEJM/Lancet PDFs don't

---

## Conclusion

v4.3.5 achieves **90.8% verified CI completion** on 43 curated RCT PDFs with 100%
extraction rate. The proximity search is now trustworthy (false positives eliminated).
The main remaining gap is extraction from complex journal layouts (NEJM, Lancet).
